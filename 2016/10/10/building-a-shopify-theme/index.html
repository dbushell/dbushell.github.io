Setup and tools

Theme development for Shopify is well documented. The Liquid reference is all I‚Äôve needed to see what data objects and filters are available for each template.

I find it best to layout the initial templates and includes with placeholder content. Once the structure is in place I go back and integrate the real data.

Shopify themes comprise of six directories:

  • /assets/ ‚Äî a flat directory of all images, CSS, and scripts
  • /templates/ ‚Äì Liquid templates for all predefined Shopify pages
  • /layout/ ‚Äì high level templates (often just one required)
  • /snippets/ ‚Äì includes that can be reused across templates
  • /config/ ‚Äî JSON data for configuration and theme settings
  • /locales/ ‚Äì locale files for internationalization

Themes have to be developed and tested on the Shopify servers. This is not ideal but they do allow the creation of dev stores. A tool I‚Äôve found useful is grunt-shopify. It fits into my existing workflow and has a watch command to upload template changes. Shopify will even return Liquid validation errors as terminal notifications; no need to refresh the browser.

Interactive baskets

When a customer clicks the ‚ÄúAdd to Basket‚Äù button Shopify will take them to the checkout page by default. Shopify does however provide a JavaScript API that makes on-page interactivity a breeze to implement.

For this project I used Cart.js which is an API wrapper and library to manage products in the basket. The trick is to override the form submission and call CartJS.addItem() passing the product variant ID, quantity, and callbacks. The Shopify API returns product data or a variety of errors such as insufficient stock so the customer can be informed accordingly.

I‚Äôm using Mustache.js for HTML templating. When Cart.js updates ‚Äî new products added or items removed ‚Äî the on-page basket is re-rendered. To do this I simply pass Mustache the HTML template and JSON data for the products.

The basket template exists within the document source for initial reference. An abbreviated example would look like this:

<script id="basket-template" type="text/x-mustache-template">
  {{# items }}
  <article class="basket-item">
    <a href="{{ url }}">{{ title }}</a>
  </article>
  {{/ items }}
  <!-- continued... -->

This method avoids heavy, manual DOM manipulation. Mustache generates all the HTML and updates a single element. For example:

var template = $('#basket-template').html();
var html = Mustache.render(template, { 'items': CartJS.cart.items });
$('.basket-popup').html(html);

If the basket design changes we can just edit the template and not worry about the JavaScript. An event listener is bound to the root basket element for the ‚Äúremove‚Äù buttons. On click the Cart.js items are updated which triggers the basket to be rendered again. Mustache is under 10kb; we don‚Äôt need anything large like Angular or React to solve this problem.

Say for example you have a ‚ÄúSmall/Red‚Äù sweater, but ‚ÄúMedium/Red‚Äù is out of stock, and the ‚ÄúLarge‚Äù size doesn‚Äôt even come in that colour. In the original single input the ‚ÄúMedium/Red‚Äù option is disabled and ‚ÄúLarge/Red‚Äù doesn‚Äôt exist.

Our new interface doesn‚Äôt account for this and it becomes quite the nightmare to figure out which options to disable. If the user selects ‚ÄúMedium‚Äù we could disable the ‚ÄúRed‚Äù option. But what if the user chooses the colour ‚ÄúRed‚Äù first, do we disable the ‚ÄúMedium‚Äù option, even if that‚Äôs their preferred size?

Amazon.co.uk disables options from top to bottom. So in this example only colours available in the selected size are enabled. If a different colour is selected, the chosen size may be reset to nothing. This requires trial and error from the users to figure out which variants exist.

I‚Äôve opted not to disable anything for simplicities sake. If users does try to add an unavailable variant to the basket they‚Äôll get an appropriate alert. This decision is helped by the fact that this store wont be managing stock levels on Shopify.
